Customer Portal Design System • v3.2

1. IT Principles

All UI decisions trace back to these core principles:

Principle Meaning UI Implication
User-firstReduce cognitive loadNext action is obvious
Autonomy by designEnable self-serviceNo support dependency
Process-firstReflect real workflowsBrowse → Select → Action
IntegratedConsistent patternsSame components everywhere
Speed with controlFast & predictableSafe, reversible actions

2. Approach

Desktop-first, responsive

Design for desktop as the primary experience. Layouts reflow gracefully on smaller screens.

Clean, utilitarian aesthetic

Customer portals should feel calm, professional and easy to interpret. Use borders and spacing (not colour) to create structure.

Neutral-dominant (95% rule)

Most of the UI uses neutral colours. Reserve colour for primary actions, active states, focus rings, and semantic status.

Small radii, crisp lines

Use small radii (4–6px) on cards, buttons and table containers. Prefer subtle borders and minimal elevation.

21. Accessibility

Make sure all UI elements meet WCAG AA contrast ratio (4.5:1 for text, 3:1 for UI elements) and are navigable via keyboard. Avoid using colour alone to convey meaning.

Minimum 36×36px (compact) or 44×44px (comfortable).

22. Notifications

Notifications keep users informed without derailing work. They must be actionable: either a click-through to the relevant record/page or an inline action (e.g., “Review request”).

Types

  • Toast: transient confirmations for non-critical actions (e.g., “Details saved”).
  • Notification Center: bell icon → aggregated list, with read/unread states and timestamps.
  • Actionable notifications: approval/authorization events that include inline actions.

Rules

  • Clickable by default: clicking takes the user to the target record/area.
  • Action-in-place when needed: join requests, approvals, authorization prompts get inline CTA buttons.
  • Respect attention: don’t spam; group similar events; provide notification settings.
  • Scope clarity: notification center is global (system scope). Page alerts remain in-page (context scope).
  • Accessibility: keyboard navigation, focus states, Esc closes panel, ARIA labels on bell/badges.

Example: actionable notification items

Your order has been dispatched
Order #ORD‑2026‑0891 • 2m ago
Joe Bloggs requests to join your organisation
Authorisation request • 1h ago
Invoice overdue
INV‑2026‑0156 • Yesterday

“Review request” should open a modal or take the user directly to the request screen with approve/deny actions.

23. Customer Onboarding

Onboarding should be step-based, low-friction, and explicit about outcomes. It must support company association (join an existing organisation vs create a new one) and handle authorisation requests cleanly.

Recommended flow

  1. Choose account type (customer/admin/supplier if applicable).
  2. Enter contact + company details with inline validation and helpful hints.
  3. Company association: match existing organisations; allow create new.
  4. Join request: if joining an existing org, show request summary + clear CTAs.
  5. Portal landing: show status (e.g., awaiting approval) and what’s available now.

Rules

  • Use a stepper/progress indicator (users always know where they are).
  • Errors appear next to fields; success confirmations can be toasts.
  • Join requests must have a “Review request” path for admins and a clear “Continue to portal” path for requesters.
  • Allow cancel/save-and-resume.
