Vanguard UI Design Guidelines

Codifying the UI/UX patterns from our prototype into a reusable design system for customer portal + website experiences.

Principles

  • Predictable structure: keep the same page “rows” in the same order so users can scan quickly.
  • Scope separation: system-level controls belong higher than page-level; page-level higher than dataset-level.
  • Neutral-first: 95% of the UI is neutral; colour is for states and emphasis only.
  • Action clarity: CTAs should be discoverable by placement + styling without competing with utilities.
  • Speed over decoration: compact, crisp controls; reduce “chrome”; keep interactions tight.

These guidelines focus on structure, layout and interaction patterns. Colour + typography are already covered and remain unchanged.

Page hierarchy pattern (stacked rows)

Think of each page as a stack of horizontal “system rows”, each with one job. Keep the same order and responsibilities, and the UI stays predictable even as modules change.

Row responsibilities

  • Main header (Context): where am I, who am I, what mode am I in. Avoid operational actions.
  • Primary nav (Global): modules + global search + settings. Always present.
  • Metrics: scannable, read-only KPIs. Neutral unless alerting.
  • Secondary + actions (Local): page state + 1–2 primary actions. System/maintenance actions separated from CTA.
  • Work area (Dataset): table tools + dataset search + view/columns controls tightly attached to the table.

Navigation

Primary navigation

  • Lives in the global control bar (yellow row).
  • Uses a subtle pill style; selected state uses primary colour fill (prototype convention).
  • Keep items short and consistent; avoid mixing actions with navigation.

Secondary navigation

  • Uses the pill switcher style (distinct from primary nav).
  • Represents a local state of the page (e.g., Records vs Receipts), not a module change.
  • Should sit on the same row as primary actions.

Search

Global search

  • Always lives in the primary nav row and searches the whole system.
  • Use ⌘ K pattern for quick access (command palette recommended).

Dataset search

  • Always lives in the table menu row and is scoped to the current dataset.
  • Never duplicate global search inside table menus.

Action hierarchy

Right side = “do the thing”

  • Primary CTA (1–2 max) lives on the right of the local action bar.
  • CTA styling uses .btn-primary to stay visually dominant.

Left side = context + maintenance

  • Secondary menu lives left.
  • System/maintenance actions (sync/import/update ERP) live left and should not compete with CTA.

Notifications

Notifications are part of the customer portal operating model: they surface time-sensitive events and requests. A notification should be actionable: clicking it should take the user to the relevant location, or offer an inline action (approve/review/pay).

Rules

  • Clickable by default: the whole item is a target (not just tiny links).
  • Always includes a “why”: title + short context line (ID/status) + timestamp.
  • Inline actions are optional: only when it prevents a trip (e.g., “Review request”).
  • Don’t bury critical states: warnings/overdue should use semantic icon + subtle background only inside the panel.
  • Batching: group by context (Orders, Access, Billing) with section labels when volume is high.

Customer onboarding

Onboarding should be step-based, low-friction, and explicit about outcomes. It must support company association (join an existing organisation vs create a new one) and handle authorisation requests cleanly.

Recommended flow

  1. Choose account type: customer, admin, or other.
  2. Enter contact and company details: with inline validation and helpful hints.
  3. Company association: match existing organisations or allow creating a new one.
  4. Join request: show request summary and provide clear CTAs for the requester (continue to portal) and administrators (review/approve).
  5. Portal landing: show current status (e.g., awaiting approval) and what’s available to the user now.

Rules

  • Use a stepper or progress indicator so users know where they are in the flow.
  • Show errors next to fields; success confirmations can use toasts.
  • Provide a clear path to review join requests for admins and a continue option for requesters.
  • Allow cancel or save-and-resume to reduce friction.

Tooltips

Tooltips provide micro-help for an icon, label, or control. They should not contain workflows, multiple links, or long content (use a popover instead).

When to use

  • Explain an icon-only control (e.g., “Columns”).
  • Clarify a data label (e.g., “Received % is based on expected quantity”).
  • Show an abbreviated value (e.g., truncated IDs).

Placement rules

  • Default placement: top (above the trigger) with an arrow.
  • If space is limited: auto-flip to bottom/left/right (implementation detail).
  • Never cover the trigger; keep a small offset (8px).
Net orders

Accessibility: tooltips must be available via keyboard focus, and trigger elements must have clear aria-label or visible text.

Popovers

Popovers are for lightweight interactive content anchored to a trigger: short forms, filters, contextual actions, or previews. If it blocks the whole page or requires a decision, use a modal.

Rules

  • Contain a title when content is more than one line.
  • Keep actions to 1–3 and align them to the right.
  • Click outside closes; Esc closes; focus is trapped only for modals (not popovers).
Filters

Popovers should never be used for long content or multi-step flows.

Bulk actions

  • Bulk toolbar appears only when rows are selected.
  • Actions are contextual to selection and grouped on the right.
  • Show count + scope on the left.

Density

  • Default: comfortable spacing for customers.
  • Provide compact mode in tables for power users where needed.
  • Never reduce hit targets below 36px for touch.

Accessibility

  • Focus states are required for all interactive controls.
  • Don’t rely on colour alone to indicate status; use icons + labels.
  • Tooltips must work on keyboard focus; popovers must have close affordances + Esc.
  • Buttons and links must have descriptive text or aria-label.
